# 体系认证-材料归档环节需求待确认问题清单

本文基于现有《体系认证-需求规格说明书0420 (1).docx》、项目协作规则及当前拟调整流程整理，用于开发前确认业务口径，避免流程、状态、数据来源在实现阶段返工。

## 一、已发现的需求冲突

| 序号 | 位置/主题 | 现有文档口径 | 当前拟调整口径 | 建议处理 |
|---|---|---|---|---|
| C01 | 归档审核任务列表-退回处理 | 二次评定退回后，项目管理人员重新提交需回到初次评定 | 二次评定退回后直接退回项目管理人员；整改后直接提交至二次评定复核，不再流转至初次评定 | 修改所有“二评退回后回初评”的描述、状态流转图、状态触发条件和异常处理 |
| C02 | 项目状态 | 文档主要以任务状态为主，未明确多体系部分通过时项目总状态 | 多体系中部分体系通过、部分仍有问题时，项目总状态为“评定中（部分通过）” | 补充项目级状态和体系级状态的关系 |
| C03 | 认证领域示例 | 多处仍写 Q/E/S/IT、ITS、信息安全体系 | 本批统一模拟 Q/E/S/F，F 为食品安全体系 | 全文替换示例，保留“动态配置”时也不能以 IT 作为默认模拟值 |
| C04 | 材料总数 | 多处写 44 项、通用材料 4 项等 | MSF11-20 权威清单为 48 项 | 全文改为 48 项，并按材料清单数据源生成统计，避免手写数量 |
| C05 | 提交按钮文案 | 归档提交页写“材料报送” | 主按钮应为“提交初次评定” | 修改按钮文案、提示语和校验说明 |
| C06 | 二次评定列表 Tab | 表格中出现两个“已退回”，且交互里提到“待复核”但 Tab 表未列出 | 二评应包含待评定、进行中、已退回、待复核、已通过/历史等明确状态 | 统一 Tab 和状态枚举 |
| C07 | 文件级评定状态 | 二次评定详情里提到具体材料带“评定状态”、文件列表含流转操作状态 | 项目规则要求评审对象是材料清单项，不对单个具体文件给结论 | 调整为材料清单项级问题和体系结论，不在文件明细表展示评定状态 |

## 二、建议的新主流程口径

| 阶段 | 入口/触发 | 处理角色 | 通过后 | 退回后 |
|---|---|---|---|---|
| 归档材料提交 | 项目完结后生成归档任务，材料齐全后提交 | 项目管理人员 | 进入初次评定-待评定 | 不适用 |
| 初次评定-专家审核 | 初评调度员分配专家 | 初评专家/初评调度员 | 进入初次复核或初评通过口径待确认 | 退回项目管理人员 |
| 初次复核 | 项目管理人员处理初评退回后重新提交 | 初评调度员/专家 | 进入二次评定 | 再次退回项目管理人员 |
| 二次评定-专家审核 | 初次复核通过后生成二评任务 | 二评专家/二评调度员 | 进入证书决定或后续节点 | 退回项目管理人员 |
| 二次评定复核 | 项目管理人员处理二评退回后重新提交 | 二评调度员/专家 | 通过体系进入下一节点；未通过体系继续处理 | 再次退回项目管理人员，不回初评 |

待确认点：你描述中“初次评定 → 分配专家审核 → 退回调整 → 初次复核 → 复核通过后进入二次评定”，建议明确首次初评如果无退回，是“专家审核完成后直接初评通过进入二评”，还是也必须经过一个“初次复核/管理员复核”动作。

## 三、逆向流程与异常场景待确认表

| 优先级 | 场景 | 当前建议规则 | 需要确认的问题 | 需确认人/系统 |
|---|---|---|---|---|
| P0 | 初评退回后重新提交 | 回到初次复核，复核范围默认仅检查退回问题和变更材料 | 初评退回整改后是否必须重新分配专家？还是沿用原专家/调度员复核即可？ | 业务负责人、评定负责人 |
| P0 | 二评退回后重新提交 | 进入二次评定复核，不回初评 | 二评复核是否必须沿用二评专家？是否允许调度员直接复核通过？ | 业务负责人、二评负责人 |
| P0 | 多体系部分通过 | 体系级可分别通过；项目级保持“评定中（部分通过）” | 已通过体系是否锁定不可再退回？未通过体系整改时是否允许影响已通过体系结论？ | 业务负责人、开发负责人 |
| P0 | 通用材料被二评退回 | 通用材料影响其覆盖的多个体系 | 如果通用材料覆盖 Q/E/S/F，但只有 Q 专家提出问题，是只退 Q，还是所有覆盖体系都变为待处理？ | 业务负责人 |
| P0 | 二评新增问题 | 二评可新增问题并退回项目管理人员 | 新增问题是否需要区分“初评漏审问题”“二评新增问题”？是否影响初评绩效/记录？ | 质量/评定负责人 |
| P0 | 证书决定退回 | 文档当前写证书决定退回后回证书决定 | 证书决定是否仍在本次范围？若退回涉及材料或审核问题，是回项目管理人员后提交到证书决定，还是回二评复核？ | 业务负责人 |
| P1 | 重新提交无变更 | 系统拦截或强提示 | 对“补充审核”类问题，可能不是上传文件变化，系统如何判断已整改？是否允许填写整改说明后提交？ | 业务负责人、开发负责人 |
| P1 | 专家超期未完成 | 提醒/催办/改派 | 超期后是否允许调度员撤回专家并重新分配？原专家已登记的问题如何保留？ | 评定负责人 |
| P1 | 专家已提交后发现错误 | 需要撤回/重新打开 | 专家提交结论后，调度员是否可退回给专家修改？是否记录版本？ | 评定负责人 |
| P1 | 退回次数不限 | 保留历史记录 | 是否需要达到 N 次退回后触发主管审批或风险提示？ | 业务负责人 |
| P1 | 项目管理人员处理退回 | 只能处理关联问题项 | 是否允许补充非退回项材料？如果补充后引发新风险，初评/二评复核是否可新增问题？ | 业务负责人 |
| P1 | 多人同时操作 | 后端按状态版本号拦截 | 是否需要乐观锁字段，如 taskVersion/statusVersion？冲突提示文案由谁定？ | 开发负责人 |
| P2 | 已通过历史保留 | 文档写默认列表移除，保留 30 天 | 保留 30 天是否真实规则？是否要支持历史查询和导出？ | 业务负责人、运维/档案负责人 |

| 前序带入文件 | 归档详情文件列表 | 认证申请、合同、审核任务、审核计划/报告、在线表单、历史附件 | 每一类 MSF11-20 材料具体从哪个业务环节/表单字段带入？是否需要映射表？ | 开发无法自动聚合 |
| 在线表单 | 文件明细预览 | 表单系统/业务表 | 在线表单是否生成 PDF 快照？评审时看实时数据还是提交时快照？ | 追溯和版本争议 |
| 本地上传文件 | 缺失材料补充 | 文件服务 + 归档材料关联表 | 上传是否自动进入资料库？是否需要审核/病毒扫描/格式转换？ | 文件安全与重复存储问题 |
| 资料库文件 | 从资料库选择 | 用户资料库/企业资料库 | 资料库是个人、部门、机构还是客户维度？谁可见、谁可维护？ | 权限泄露或找不到文件 |

| 问题清单 | 退回、复核、统计 | 专家/调度员登记 | 问题必须关联材料项，是否还要关联具体文件？问题状态有哪些终态？ | 复核范围不清 |
| 整改记录 | 项目管理人员重新提交 | 整改说明 + 文件变更记录 | 每次重新提交是否必须填写整改说明？是否按问题逐条填写？ | 复核无法判断整改 |
| 专家信息 | 分配弹窗、专家工作台 | 专家库/人员资质库 | 专家库由哪个系统维护？字段包括体系授权、专业代码、区域、机构、内外部、有效期吗？ | 分配不符合资质 |
| 专家负荷 | 分配预警 | 工作流待办/任务表 | “当前承接 5 个任务提醒”的统计范围是全部任务、同阶段任务还是同周期任务？ | 负荷预警失真 |
| 专业代码 | 二评专家覆盖 | 项目认证范围 + 专家资质 | 二评必须覆盖专业代码，代码来自申请、审核方案还是证书范围？ | 二评无法判定可通过 |
| 通知消息 | 分配、退回、催办 | 消息中心/短信/站内信 | 哪些节点发站内信、短信或邮件？失败是否重发？ | 用户不知道待办 |

## 五、建议补充的状态模型

### 1. 项目级状态

| 状态 | 触发条件 | 说明 |
|---|---|---|
| 归档待提交 | 材料任务生成但未提交初评 | 项目管理人员处理 |
| 初次评定中 | 已提交初评，初评专家/调度员处理中 | 包含待评定、进行中 |
| 初次评定退回 | 初评不通过 | 项目管理人员整改后回初次复核 |
| 初次复核中 | 初评退回整改后重新提交 | 通过后进二评 |
| 二次评定中 | 初次复核/初评通过后进入二评 | 包含待评定、进行中 |
| 二次评定退回 | 二评不通过 | 项目管理人员整改后回二次复核 |
| 二次复核中 | 二评退回整改后重新提交 | 不回初评 |
| 评定中（部分通过） | 至少一个体系二评通过，至少一个体系待处理/退回/复核中 | 项目未整体通过 |
| 证书决定中 | 全部体系二评通过后进入 | 后续节点 |
| 已完结 | 证书决定通过/流程终态 | 只读 |

### 2. 体系级状态

| 状态 | 说明 |
|---|---|
| 待评定 | 该体系尚未开始当前阶段评定 |
| 评定中 | 专家或调度员正在处理 |
| 已退回 | 该体系存在需项目管理人员处理的问题 |
| 待复核 | 项目管理人员已提交整改，等待对应阶段复核 |
| 已通过 | 该体系当前阶段已通过 |
| 已锁定 | 可选状态：二评通过后不允许项目管理人员再改动该体系材料 |

## 六、开发前建议必须确认的关键问题

| 编号 | 问题 | 建议答案方向 |
|---|---|---|
| Q01 | 初次评定首次专家审核完成后，是否还必须有“初次复核”动作？ | 建议：初评退回后的整改才叫初次复核；首次无问题可由调度员确认后直接进入二评 |
| Q02 | 二次评定退回整改后，谁来复核？ | 建议：进入二次评定复核，由二评调度员主责，可按配置决定是否再次派专家 |
| Q03 | 分体系通过后，已通过体系材料是否锁定？ | 建议：锁定已通过体系相关材料；如通用材料被修改，提示可能影响已通过体系并要求重新确认 |
| Q04 | 通用材料问题影响哪些体系？ | 建议：问题登记时必须选择涉及体系；默认不因通用材料自动污染全部体系，由评审人选择影响范围 |
| Q05 | “适用时必须上报”的适用条件谁判断？ | 建议：先提供人工“适用/不适用”确认字段和原因，后续再规则化 |
| Q06 | 专家选择数据源在哪里维护？ | 建议：独立专家库，至少维护体系授权、专业代码、有效期、区域/机构、负荷、状态 |
| Q07 | 退回问题是否必须逐条整改？ | 建议：项目管理人员按问题逐条填写整改说明，并可关联新增/替换文件 |
| Q08 | 证书决定退回是否纳入本次调整？ | 建议：如纳入，需明确退回到项目管理人员后回证书决定、二评复核，还是按退回原因分流 |
