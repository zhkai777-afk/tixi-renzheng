# 体系认证项目协作说明

## 每次修改前必须读取

在优化本项目任意页面前，先读取以下文件：

1. `PROJECT_REQUIREMENTS.md`：项目角色、流程、状态流转和关键业务规则。
2. `page-style-spec.md`：统一页面布局、视觉风格和交互规范。
3. `materials-catalog.json`：MSF11-20 审核材料归档/上报清单权威数据源。
4. `二次评定-详情.html`：当前项目 UI 风格基准页面。

## 统一业务设定

- 模拟项目认证领域统一使用 `Q/E/S/F`。
- 体系含义：
  - `Q`：质量管理体系
  - `E`：环境管理体系
  - `S`：职业健康安全
  - `F`：食品安全体系
- 不再使用 `IT` 作为本批页面的默认模拟体系。
- 材料清单统一使用 MSF11-20 的 48 项材料，不得再手写 44 项、29 项等不一致数量。

## 页面优化原则

- 整个页面风格都要对齐 `二次评定-详情.html`，不是只优化材料清单局部。
- 保持后台业务系统风格：清晰、克制、信息密度高、无营销化装饰。
- 优先使用白色面板、浅灰背景、细边框、8px 圆角、13-14px 业务文本。
- 体系徽标、状态标签、表格、筛选、弹窗、右侧信息面板等都应使用统一视觉规则。
- 页面中如需模拟数据，应服务于真实业务流程，不做与流程无关的装饰性内容。
- 页面整体区域间距参考 `初次评定-详情-优化版.html`：页面外层、顶部信息区、左侧材料栏、中间工作区和右侧控制台之间应有明确留白，避免模块贴边或连成一整片。
- 页面需要自适应浏览器窗口宽度，不再默认依赖固定 `1848px` 页面宽度；优先使用 `100vw`、`max-width: 100%`、`grid`、`clamp()`、`minmax(0, 1fr)` 和必要断点保证三栏布局可随窗口收缩。

## 页面优化协作流程

后续优化任意页面时，默认先分析并输出调整方案，等用户确认后再进行代码调整；除非用户明确要求直接修改。

分析阶段必须包含：

1. 当前页面的业务角色和使用场景：先判断页面由谁使用，例如项目管理人员、初次评定管理员、二次评定管理员或专家。
2. 当前用户进入页面后的核心目标：明确用户最关心的是补齐材料、判断是否可提交、指派专家、处理问题、通过/退回，还是二次评定放行。
3. 页面重点信息优先级：按“决策所需信息优先”的原则排序，不按现有 DOM 或视觉顺序照搬。
4. 当前页面结构问题：指出顶部、左侧材料树、中间主工作区、右侧控制台、弹窗和交互反馈中影响判断效率或风格统一的点。
5. 调整方案：输出清晰的布局设计思路、UI 优化方向、信息层级和交互规则。
6. 实施顺序：说明先改整体布局，再改核心业务模块，最后统一细节和校验。

确认后实施阶段遵循：

- 如用户要求“复制一份后修改”，必须先复制原文件，在副本上调整，原文件不动。
- 每次改动都围绕已确认方案执行，不擅自扩展无关功能。
- 优先保留真实业务链路和已有可用逻辑，必要时再重构样式和渲染结构。
- 调整完成后做静态检查：不得出现 `IT` 默认体系、不得使用浏览器默认 `alert` / `confirm`、不得将具体文件作为评定结论对象、不得破坏 MSF11-20 的 48 项材料口径。

## 页面布局设计方法

后台详情页统一采用“顶部任务信息 + 左侧材料导航 + 中间主工作区 + 右侧控制台”的决策型布局。不同页面应根据角色调整信息重点，但布局语言保持一致。

整体布局：

- 页面外层建议使用浅灰背景、`12-16px` 内边距和 `12-16px` 模块间距；顶部、左侧、中间、右侧区域应作为独立白色面板呈现，使用细边框和 8px 圆角。
- 主体三栏优先使用响应式 `grid` 布局：左侧材料栏约 `280-320px`，右侧控制台约 `320-380px`，中间主工作区使用 `minmax(0, 1fr)` 自适应。
- 窗口宽度不足时应通过 `clamp()` 和媒体查询收缩左右栏，必要时将右侧控制台下移或改为横向面板，避免内容被固定宽度挤出窗口。
- 顶部任务信息允许换行或压缩，不能因任务号、客户、体系等信息撑出横向滚动。

顶部区域：

- 展示返回、页面标题、任务号、客户、Q/E/S/F 项目概况、当前状态和主操作。
- 如果页面需要快速判断整体状态，可在顶部加入四体系状态卡或摘要卡，但样式应克制，使用白底、细边框和浅色状态标签。
- 顶部提示文案应说明当前页面的下一步业务动作，不写泛化说明。

左侧材料导航：

- 统一使用 MSF11-20 48 项材料清单，按“通用材料 / 一阶段材料 / 二阶段材料”分组。
- 第一项为“全部材料跨体系视图”，样式与普通材料项一致。
- 材料项第一行只显示材料名称；体系徽标单独一行；底部显示分类/适用说明、必交标签、文件数或问题数。
- 筛选项根据页面角色命名：归档提交页可用“全部/缺失/已带入”，评定页可用“全部/有问题/无问题/无文件/待评定”等。

中间主工作区：

- 按当前角色的关键判断组织内容，不机械复用其他页面顺序。
- 归档提交页重点是完整性和文件补充：可先显示体系完整度，再显示材料明细表。
- 初次评定、二次评定、专家评审页重点是问题和结论：优先显示体系/任务状态、问题清单，再显示材料明细表。
- 材料明细表固定为“具体文件名称 / 适用体系 / 操作”三列；文件来源、上传时间、类型等放在文件名下方辅助信息中。
- 评定状态和问题必须落在材料清单项级别，不在具体文件行上展示“评定状态”列。

右侧控制台：

- 用于承载当前任务状态、阻断原因、下一步动作和关键统计。
- 初次评定页重点展示专家覆盖、专家评定进度、有效问题、通过/退回条件。
- 二次评定页重点展示专家覆盖、体系级分流、可通过体系、需退回体系和问题清理情况。
- 归档提交页重点展示各体系完整性、缺失项和提交条件。
- 主按钮禁用时必须给出明确原因，不能只显示不可点击。

## UI 优化经验

- 先解决信息架构，再做视觉美化；如果页面重点不清晰，单纯调颜色和间距不会真正改善体验。
- 页面标题、按钮和状态文案必须使用业务语言，如“通过初评”“退回任务”“提交初次评定”，避免“确认”“处理”“操作”等泛化词单独出现。
- 体系颜色只用于 Q/E/S/F 标签、状态辅助和轻量强调，不做大面积高饱和色块。
- 避免营销化视觉：不使用大渐变背景、装饰图形、夸张阴影、过大的标题字号。
- 表格和面板保持高信息密度：13px 表格文本、14px 重点文件名、15px 面板标题、8px 圆角。
- 弹窗、toast、删除确认、提交确认都使用自定义组件，不使用浏览器默认弹窗。
- 空状态应说明缺什么、为什么影响流程、下一步如何处理。
- 模拟数据必须服务于真实流程，例如专家覆盖、有效问题、材料缺失、退回原因；不要添加与流程无关的装饰数据。

## 归档材料流程要点

归档材料提交页由项目管理人员使用。系统默认带出前序业务环节产生的文件；缺失材料可通过本地上传或资料库选择补充；所有体系必备材料齐全后，主按钮启用，按钮文案为“提交初次评定”。

二次评定通过后，本阶段业务结束并进入已完结；二次评定退回后，项目管理人员补充材料再次提交时直接回到二次评定，不再经过初次评定。
